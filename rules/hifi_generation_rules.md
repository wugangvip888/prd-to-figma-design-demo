# Hi-Fi Generation Rules

## 目标

定义根据 Visual Spec 生成 Figma 高保真设计稿的执行规则。

## 执行起点（强制）

```text
高保真生成必须从已审核线框图 Frame 复制开始，不得从空白 Frame 或空白页面新建结构。

标准执行步骤：
1. 读取 workspace/figma_targets.md 确认线框图 Frame ID 和高保真输出端页面
2. 用 use_figma 将线框图 Frame 完整复制到高保真输出页面
3. 在复制后的 Frame 上叠加视觉属性（填充/描边/圆角/阴影/字体样式/资产替换）
4. 不得改变线框图的节点结构、层级关系、绝对坐标和命名
```

## 逐模块执行前置声明（强制）

```text
每个模块开始执行前，必须完成以下三步声明，未声明不得动 Figma：

### Step A：读取目标节点实际结构

操作类型不同，读取对象不同，不得互换，不得跳过：

- Clone / 复用操作前：
  必须用 use_figma 读取【线框图】对应模块的完整子节点树（id、name、type、x、y、w、h、chars），
  不得基于记忆、文档描述或上一轮执行经验假设线框图中存在或不存在某个节点。
  理由：线框图是执行起点，其实际结构决定哪些节点需要保留、删除或新增。

- 仅叠加视觉样式（fills/strokes/cornerRadius/shadow/font）时：
  只需读取目标节点的 fills、strokes、effects、cornerRadius，不需要读全量子节点树。

### Step B：声明本模块受哪些规则约束

在执行前，必须显式列出本模块涉及的规则条目（不是重读规则，是绑定声明）。
常见规则条目包括但不限于：
- Module 背景与内容边距规则（凡 Module 有背景属性时必选）
- Clone 后强制步骤 Level 1 或 Level 2 顺序（凡有复用操作时必选）
- 横向溢出执行规则（凡有横向排列子元素时必选）
- 固定底部 CTA 规则（凡处理 CTA 模块时必选）
- 新建 TEXT 节点强制核查清单（凡有新建文字节点时必选）

未声明的规则条目，在执行中违反时视为「规则遗漏」，不得以「不知道」为由免责。

### Step C：对照 Visual Spec「不继承」列表做 diff

将 Step A 读出的线框图子节点列表，逐一对照 Visual Spec 中该模块的「不继承」列，
明确标出哪些节点需要删除、哪些需要新增、哪些只修改样式。
输出格式：
- 保留并改样式：[节点名] [id]
- 删除：[节点名] [id]（原因：Visual Spec 不继承 / PRD 未定义）
- 新增：[节点名]（原因：Visual Spec 要求 / 等价替换）

未输出此 diff 列表，不得进入执行步骤。
```

## Auto Layout 边界（强制）

```text
高保真阶段不引入 Auto Layout。
保持线框图的静态 Frame + 绝对坐标结构，仅在原有结构上叠加视觉样式。
Auto Layout 结构化工作在设计层（Flow 09）阶段执行，不在高保真阶段执行。

唯一例外：Level 1 直接复用采样源组件时，如采样源组件本身为 Auto Layout 结构，
允许整体复用其 Auto Layout，但必须在 run record 中记录复用来源节点 ID 和理由。
```

## 基本原则

```text
高保真 Page Frame 尺寸固定为 360 x 780。
PRD 中出现的设备基准、手机型号、pt 尺寸、倍率、安全区或适配说明，只作为业务与适配参考，不得覆盖高保真 Page Frame 尺寸。
高保真以已审核线框图为第一参考；必须一一对应线框图的页面数量、Frame 顺序、模块层级、模块包含关系、可见文案、状态表达、CTA 位置和控件相对位置。
PRD 与 PRD 补充记录是业务硬边界；Intent 用于校验页面目标、用户首要动作和视觉重心；Priority Map 用于校验 P0/P1/P2/P3 视觉层级；Layout Spec / Visual Spec 只作为结构和视觉执行说明，不得推翻已审核线框图。
生成结果必须完整显示线框图内容，不得增加线框图没有的业务模块、按钮、状态或文案，也不得遗漏线框图已有内容。
08 阶段不得临场改写业务结构、功能范围、页面状态和核心设计规则。
```

## 字体执行规则

```text
Hi-Fi Generation 阶段不处理最终字体替换。
新建文字节点默认使用 Figma 可用的 Inter。
不得主动加载、匹配、替换或修复字体。
不得因为字体缺失、字体权限、字体不可加载或字体不一致而重试。
如果采样源字体不可用，保留当前可用字体并继续执行。
最终字体由人工在流程完成后统一替换。
```

## 视觉采样优先级

```text
采样端 Figma 来源 / Design System 只用于视觉样式、组件风格、间距、字体、颜色、按钮样式和可编辑图层结构参考，不得改变业务结构。
视觉采样只能迁移样式方法、可复用视觉资产和组件外观，不得迁移目标线框图没有的内容模块、Tab、按钮、状态、字段、交互控件或业务入口。
当采样源组件包含目标页面没有的子结构时，只能提取其颜色、字体、圆角、描边、阴影、图片 / 图标资产和排版规律；不得把该子结构直接复制到目标页面。
严禁把采样稿、旧界面或采样端 Figma 来源的页面内容整体复制、整体移植或整体替换到当前设计稿；采样稿只能作为局部样式、组件外观、资产和排版规律的参考。
严禁把采样稿、旧界面或采样端 Figma 来源的业务文案迁移到当前设计稿；文案、功能架构、业务状态和交互语义必须以当前目标线框图、Visual Spec、PRD 或人工确认内容为准。
允许在功能语义等价的前提下复用采样源组件、组件结构模式、布局模式和交互控件形态；组件形态可以变化，功能属性不能变化，视觉结构可以重组，业务语义不能重写。
允许的等价视觉表达替换包括但不限于：文本返回替换为返回图标；文字按钮、图标按钮、图标 + 文字按钮之间互换；Tab 下划线、胶囊、分段控件等视觉样式互换；checkbox / switch、radio / segmented control、dropdown / bottom sheet / picker 等控件表现互换。
等价替换必须保持控件代表的业务对象、触发动作、可选项含义、选项数量、默认选中 / 激活 / 禁用状态语义、输入字段数据含义、必要入口和控件命名可追溯性；不得引入新业务入口、不得删除目标稿已有功能、不得迁移采样稿文案。
高保真生成必须执行 `rules/design_system_rules.md` 中的复用分级：三维判定框架（维度A结构组成模式 + 维度B action类型 + 维度C视觉结构模式），复用是默认行为，Level 3 重建必须逐条写出三个维度均无法匹配的原因。不得把 Level 1 / Level 2 的 source_candidate 仅当作 token 参考后手工重绘。
复用级别判断以节点语义职责为准，不以视觉实现形式为准（语义等价原则详见 `rules/design_system_rules.md`）；Tab 样式差异（胶囊/下划线/填色块）、divider 实现差异（线/矩形/留白）等视觉形式不同但职责相同的情况，不触发降级。
Level 1 禁止任何节点增删操作，包括删除采样源中"多余"的节点。如需增删节点，必须升级为 Level 2。
状态样式只能作用于对应状态控件节点。
不得通过尺寸范围、节点类型等模糊条件定位目标节点。
必须按图层名、父子层级或 Visual Spec 中的明确节点路径定位。
如果生成结果出现采样稿页面结构、旧页面业务内容或旧界面状态整页进入当前设计稿，必须判定为严重生成偏差，立即停止继续生成并回滚该页到采样前的目标结构后再局部修正。
采样必须按 `workspace/figma_targets.md` 中采样端链接优先级执行；每行使用 `数字. Figma链接` 格式，数字越小优先级越高。
同一数字允许多条 Figma 链接，权重相同，按书写顺序读取。
采样链接可以指向 Figma 文件、Page、Section、Frame 或具体节点；流程只解析链接本身，不要求额外填写 pageID、Page 名称或 nodeID。
上一级能支撑判断时，不得跳到下一级；如果某一优先级无法访问、内容不足或无法覆盖当前模块，必须在 run record 中记录降级原因、采样优先级、采样链接和使用的下一级来源。
同层或跨层采样结论冲突时，不得自行裁定，必须记录冲突项、采样链接和节点 ID，并作为 Gate 阻塞项停止当前流程。
不得使用旧规则硬编码颜色、渐变、圆角或组件样式；所有视觉规则必须能追溯到采样来源、Design System 草案或人工确认记录。
复制渐变样式时必须同时复制 gradientStops 和 gradientTransform；不得只复制颜色值或使用 Figma 默认渐变方向。
```

## 旧文件视觉主导规则

```text
高保真阶段必须以视觉采样源文件作为视觉表现的最高优先级来源。

优先级边界如下：
1. PRD 决定文案、业务含义、字段、功能范围和模块是否存在。
2. 已审核 Wireframe 决定页面数量、模块顺序、信息层级、页面结构、核心布局和 P0/P1/P2/P3 显著性。
3. 采样端 Figma 来源 / Design System 决定视觉表现，包括字号、字重、行高、颜色、背景、描边、圆角、阴影、间距、组件尺寸、组件状态和视觉密度。

在 Hi-Fi Generation 阶段，结构已由线框图锁定；在不改变线框图结构和 PRD 业务语义的前提下，采样源视觉属性是视觉执行的最高优先级。Design System 阶段必须完整采集，Visual Spec 必须完整写入，Hi-Fi Generation 必须完整执行。任何视觉属性的缺失或替换，必须有明确的冲突原因记录，否则视为执行偏差，Gate 判定 FAIL。

当线框图组件在视觉采样源中存在相同或相似组件时，必须优先继承该组件的视觉属性与内部结构，包括：
尺寸比例、内边距、元素间距、字号、字重、行高、颜色、背景、描边、圆角、阴影、图标 / 图片资产、选中 / 禁用 / 默认状态样式。

当不存在相同组件时，必须从视觉采样源中查找相似 Frame 或相似交互控件，提取其 Frame 属性作为生成依据，包括：
页面左右边距、模块上下间距、卡片内边距、标题与内容间距、同组元素间距、对齐方式、圆角等级、描边等级、阴影等级、背景使用规则和视觉权重。

当组件级与 Frame 级均无法匹配时，必须使用 design-system-draft.json 中已有的按需 style_id 作为兜底；不得凭空创造新的颜色、圆角、阴影、字号、行高或间距层级。

当旧文件视觉属性与 PRD 文案长度或线框图结构发生冲突时：
1. PRD 文案不可改写、缩写、扩写、翻译或替换。
2. 模块存在性、模块顺序和业务层级以已审核 Wireframe 为准。
3. 视觉样式仍优先贴近旧文件。
4. 只允许为适配新文案做最小尺寸、换行或局部间距调整。
5. 调整原因必须写入 run record。
6. 不得使用人工估算 gap 重排复用组件；必须从源组件同语义节点的实际 x / y / width / height 计算间距后再应用。
```

## 横向溢出执行规则

```text
当子元素横向排布后超出父级容器或页面安全边距时，必须先判断该区域是否为明确的横向滚动 / 横滑容器。

若是横向滚动 / 横滑容器：
- 必须在 Wireframe、Layout Spec 或 Visual Spec 中有明确横滑 / 横向滚动标注。
- 没有明确标注时，默认视为非横滑容器处理。
- 父级 Frame 必须保持在页面安全边距内。
- 允许超出的只能是内部滚动内容，不得扩大父级 Frame 突破安全边距。

若不是横向滚动 / 横滑容器：
- 不得横向超出父级容器或页面安全边距。
- 必须通过换行、分行、压缩间距到源规则允许范围内、或减少单行数量解决。
- 不得通过扩大父级 Frame 或突破页面边距解决。
- 不得压缩字段名、字段值、按钮文案、标题等业务文本节点的实际显示宽度。
- 必须先保证业务文本完整展示，再通过调整兄弟元素位置、背景 shape 宽度、换行或分行解决。

参数选择器、标签组、chip group 等非横滑的横向重复控件，在设计层 / Auto Layout 阶段必须表达为可换行列表：
- 父级宽度变化时，子项应按可用宽度重新提行或换行。
- 不得只让父级 Frame stretch，而让内部 chip 保持固定绝对坐标。
- 如需新增 list / wrapper 表达 wrap，必须记录结构调整原因、静态视觉校验和拉伸测试结果。

如果某横向内容区用于静态展示横向列表，但不需要在当前状态表达完整滚动交互，可按视觉要求设置为非对称可视区：
- 起点遵循左安全边距。
- 宽度可延展到屏幕右边界，右边距为 0。
- 初始静态状态不得裁切最左侧内容。
- 内部列表可向右延展，但不得向左裁切首项。
```

## 旧稿骨架继承边界

```text
旧稿视觉骨架只在目标页面与采样源页面结构同构时允许继承。

结构同构必须同时满足：
1. 模块语义一致。
2. 模块顺序一致。
3. 主模块数量一致，或差异不影响页面整体布局。
4. 子组件类型和数量基本一致。
5. 内容密度接近。
6. PRD 原文案放入后不造成明显溢出、遮挡、错误压缩或核心信息丢失。

满足结构同构时，允许继承旧稿的模块 x/y/w/h、卡片高度、模块间距、内边距、底部 CTA 区域和图片卡尺寸。

如果不满足结构同构条件，不得强行套用旧稿的模块位置、模块高度或整页骨架；只能继承旧稿的设计规范，包括：
颜色、字号、字重、行高、圆角、描边、阴影、图片 / 图标资产、组件密度、padding、gap 和对齐方式。

当结构同构判断不明确时，必须降级为 Frame 级推导或 style_id 兜底，并在 run record 中记录原因。

不得为了贴近旧稿骨架而改变 PRD 文案、删减目标线框图模块、压缩到不可读、制造重叠、隐藏核心内容或改变业务层级。
```

## Frame 结构与命名

```text
高保真输出 Page Frame 必须与线框图 Frame 一一对应，命名保持一致：
{序号}_{page_id}_state_{state}

Page Frame 内部使用稳定可编辑结构：
Page Frame
  Module Frame
    Control Frame
      Element

Module / Control / Element 命名必须复用线框图语义命名，例如 nav_bar、creation_input_card、uploaded_file_list、uploaded_file_item_1、remove_file_action、replace_file_action、parameter_controls、parameter_edit_action、primary_cta。
不得出现 Frame 1、Group 1、Rectangle 123、Text 45、按钮1、卡片2 这类无语义命名。
高保真可以替换视觉样式，也可以在不删减功能的前提下调整组件内部表现结构（如将占位图替换为真实资产、将低保真组件升级为采样源中的成熟组件样式）；但不得改变线框图已明确的子元素上下顺序、左右顺序、父子归属和区域关系。线框图中位于图片 / 媒体区域外的文本、标签、按钮或状态，不得移入图片 / 媒体区域内部；不得为承载此类移动而新增线框图未定义的遮罩、覆盖层、角标容器或背景条。不得改变线框图的业务模块、功能属性、必要入口、选项语义和业务命名。
同一语义操作或同一语义标题只能保留一套表达；不得同时存在旧标题和新标题、文本返回和图标返回、重复按钮或重复入口。
当视觉采样或人工修改引入同义控件时，必须先判断哪一套是当前目标结构，再删除或隐藏另一套，并在 run record 中记录处理结果。

自动布局和 Frame 命名必须表达结构职责，而不是表达 Figma 操作痕迹：
1. 页面主业务区块使用 `{module}_section`，例如 creation_input_section、inspiration_section、parameter_section。
2. 仅为 Auto Layout padding、gap、alignment 或背景承载而新增的外层 Frame，使用 `{module}_container`、`{module}_wrapper` 或 `{module}_padding_wrapper`；不得保留 Frame 123 这类默认名。
3. 具备视觉背景、描边、阴影或圆角的承载层使用 `{module}_card`、`{module}_surface` 或 `{module}_panel`。
4. 横向 / 纵向重复内容容器使用 `{module}_list` 或 `{module}_carousel_list`；单个重复项使用 `{module}_item_{n}`、`{module}_card_{n}`、`{module}_chip_{semantic}`。
5. 图片、缩略图和图标等 Element 不得与父级 card / item 重名；图片使用 `{module}_thumb_{n}`、图标使用 `{action}_icon`。
6. 头部行、底部行和操作行使用 `{module}_header`、`{module}_footer`、`{module}_action_row` 或具体语义名，例如 input_footer_actions。
7. 字段名和值分离时，字段名使用 `{field}_label`，字段值使用 `{field}_value`。
8. 固定底部悬浮 CTA 使用 `{action}_fixed_cta` 或 `{module}_fixed_cta`，例如 generate_comic_fixed_cta；该层命名必须保留 fixed 语义，表示贴底悬浮，不参与普通内容流布局。
9. 不使用纯位置词、视觉词或无上下文数字作为核心命名，例如 top_group、pink_frame、card_1；数字尾缀只允许用于重复项或资产序列。
10. 命名统一使用英文 snake_case；业务语义优先，结构职责次之，视觉属性只在必要时作为后缀补充。
11. 为已有静态稿补 Auto Layout 时，必须优先保持静态图层的视觉位置、尺寸和间距；不得为了套用 Auto Layout 改动静态图效果。只有当原始位置与 Auto Layout 适配目标存在明确冲突时，才允许保守跳过局部 Auto Layout 或使用普通 Frame + constraints，并必须记录冲突点、影响范围和自动保守处理结果。
```

## 资产复用规则

```text
当采样源 Figma 节点可访问时，图标、插画、图片、装饰笔触和真实媒体缩略图必须优先复用采样源 Figma 节点或其真实图片填充。
不得在已有采样源资产可用时使用默认占位图、抽象占位插画或手工临摹图标替代。
截图只能作为视觉校验和资产不可访问时的备选依据；如果只能从截图裁剪资产，必须在 run record 中记录原因。
装饰性资产必须归属到对应语义 Frame 内部，并位于主要内容图层下方；不得脱离所属模块，也不得遮挡标题、正文、按钮或图标。
图片填充目标节点识别规则：节点命名中包含 image、thumb、cover、photo、media 任意一个关键词的矩形或 Frame，视为需要填充图片的目标节点，必须精确操作该节点，不得对其父级节点操作。
当需要填充图片时，按以下顺序提取关键词：1. 目标节点自身的文案；2. 同属父级 Frame 内的文案组合；3. 所在模块的模块名称。将提取到的文案翻译为英文后，调用 scripts/fetch_placeholder_image.py --query "英文关键词" 获取图片 URL，在 URL 末尾追加 &w=720&h=780&fit=crop 控制图片尺寸，以 Fill 模式填充到目标节点；如果调用失败，填充中性灰 #EDEDED，不强制重试。
同一批次生成中，相同语义关键词的图片只调用 scripts/fetch_placeholder_image.py 一次，返回的 URL 记录在当前执行上下文中；后续遇到相同语义的目标节点，直接复用已有 URL，不重复请求。
需要请求多张不同图片时，必须逐张串行调用，每次调用完成后再发起下一次，不得并发请求。
```

## 模块边界与间距规则

```text
用于计算页面模块间距的 Module Frame 必须收紧到有效子元素外接边界，避免把页面安全边距、辅助空白或历史残留空白计入模块内部边距。
页面主模块间距必须基于 Module Frame 的真实外接边界计算，不得基于视觉估计或散落子图层计算。
同一页面的同级主模块应使用一致的间距值；如需要例外，必须能追溯到 Visual Spec、PRD 或人工确认记录。
上游模块位置或高度变化后，下游模块必须按净边界变化同步调整，保持既定模块间距不变。
审核模块间距时必须同时检查节点结构变化；如果元素从父级内部移出成为同级节点，间距计算必须改用新的真实结构边界。
```

## Clone 后强制步骤

### Level 1 执行顺序（完全匹配，直接复用）

```text
Level 1 的核心约束：不允许任何节点增删。有增删需求即升级为 Level 2。

⓪ clone 后立即用 use_figma 读取 clonedFrameId 的基础属性（id、name、type），
   确认节点实际存在后才能继续。
   若读取返回空或报错，视为 clone 失败，停止执行并报告错误，不得继续后续步骤。

① 读线框图目标模块子节点树（来自「逐模块执行前置声明 Step A」的结果）
② 对照 Visual Spec「不继承」列表做 diff
   ——若 diff 结果出现「需要删除」或「需要新增」任何节点，立即升级为 Level 2，不得继续 Level 1 流程
   ——若 diff 结果仅为「改样式」，继续以下步骤
③ 叠加视觉样式（fills / strokes / cornerRadius / shadow / font / color）
   不得修改任何节点的 x / y / width / height
④ 替换文案（characters）
   替换前必须输出以下溯源声明，否则不得执行替换：
   「本模块所有文案来源：线框图节点 [ID] chars = "…"，字号/字重/行高来源：Visual Spec [章节号] = X px / [字重] / X px，不使用采样源文案和采样源字号。」
   未输出声明、声明中无节点 ID、或字号/字重/行高未引用 Visual Spec 章节号的，视为溯源不完整，不得继续。
   替换后读取文字节点实际 width，检查是否触发级联修正（见下方级联修正边界）
⑤ 截图确认（截图 scope 要求见「逐模块截图确认 scope 规则」）
⑥ 对照 diff 列表逐项核验：每一项「改样式」是否已执行，无遗漏

级联修正边界（仅 Level 1 允许的坐标操作，两条缺一不可）：
a. 只能修正同层级的直接兄弟节点（如 value 变宽后修正同行的 arrow x）
b. 止步于直接父容器——父容器尺寸不动，跨模块节点不动
超出此边界的任何坐标修改视为「重新定位」，必须升级为 Level 2。
```

### Level 2 执行顺序（高相似，修改后复用）

```text
Level 2 允许增删节点，但顺序必须严格遵守：先删再补再改，不得乱序。

⓪ clone 后立即用 use_figma 读取 clonedFrameId 的基础属性（id、name、type），
   确认节点实际存在后才能继续。
   若读取返回空或报错，视为 clone 失败，停止执行并报告错误，不得继续后续步骤。

① 读线框图目标模块子节点树（来自「逐模块执行前置声明 Step A」的结果）
② 对照 Visual Spec「不继承」列表做 diff，输出三类清单：
   - 保留并改样式：[节点名] [id]
   - 删除：[节点名] [id]
   - 新增：[节点名]
   diff 列表必须展开到图标级子节点。凡父级节点内包含 icon、arrow、chevron、indicator、selected_indicator 语义子节点的，必须逐一列出；凡子节点类型为 VECTOR、INSTANCE 或 BOOLEAN_OPERATION 的，无论命名如何，均视为视觉资产节点，必须列入 diff。所有视觉资产子节点须明确标注「保留并改样式」或「删除」，不得以父级代替子级。缺少以上展开的 diff 视为不完整，不得进入执行步骤。
③ 删除 diff 中标记为「删除」的节点
   必须逐一 use_figma 删除，不得跳过，不得只改 fill 掩盖（例：Tab 的 active_surface 若不继承，
   必须删除该节点，不得只把 fill 改透明或 opacity 改 0）
④ 新增 diff 中标记为「新增」的节点，按 Visual Spec 规格创建
⑤ 叠加视觉样式（保留并改样式的节点）
⑥ 替换文案，替换前必须输出以下溯源声明，否则不得执行替换：
   「本模块所有文案来源：线框图节点 [ID] chars = "…"，字号/字重/行高来源：Visual Spec [章节号] = X px / [字重] / X px，不使用采样源文案和采样源字号。」
   未输出声明、声明中无节点 ID、或字号/字重/行高未引用 Visual Spec 章节号的，视为溯源不完整，不得继续。
   读取实际 width，按「前元素 x + 前元素宽 + 采样源 gap」重算同行兄弟元素位置
   不得手写坐标，不得凭页面宽度估算
⑦ 截图确认（截图 scope 要求见「逐模块截图确认 scope 规则」）
⑧ 对照 diff 列表逐项核验：删除项已删、新增项已加、改样式项已执行，无遗漏

模块定位补充约束：
Level 2 clone 后，第一步必须读采样源节点在其父级中的实际 x/y 坐标，以此为基准定位目标节点。
不得直接手写目标坐标，不得凭页面宽度或模块顺序估算位置。
模块间距必须从采样源实际 y 坐标差值计算，写入 run record，不得估算。
```

### Level 2 同义控件强制删除（Step A 结束后、Step B 开始前）

```text
diff 列表中标注为"删除"的同义控件节点，必须在 Step B 开始前执行删除：

  执行顺序：
  1. Step A clone 完成（width 校验由 hifi_geometry gate Detection C 执行）
  2. 读取本模块 Level 2 diff 中标注为"删除"的节点 ID 列表
     （来自 Step A clone 后 diff 步骤的实时输出，非 Visual Spec 原文；
      Visual Spec 仅提供语义判断依据，不作为节点 ID 的直接来源）
  3. 对每个删除节点调用 use_figma 执行实际删除操作
  4. 调用 get_node 确认该节点 ID 已不存在（不存在 = 删除成功）
  5. 截图核验，核验项中必须包含："同义控件已删除、无残留节点"
  6. 上述步骤全部完成后，才能进入 Step B

禁止以"在 Step B 修改样式时视觉掩盖"代替删除操作。
截图核验若仍可见被删除节点的视觉痕迹，Step B 不得开始。
```

## Level 2 新增节点间距规则

```text
Level 2 补齐的新增节点（采样源无直接参考对象）的间距必须有可追溯依据。

优先级链：
1. 线框图：从线框图中读取新增节点与相邻节点的实际距离
2. Visual Spec：从 Visual Spec 的间距说明中读取
3. design-system-draft.json：从 spacing_* 条目中读取对应语义间距
4. 兜底：以上均无时，从同组件内采样，规则如下：

兜底采样规则：
a. 优先找同等级别间距：
   - 组件间距：同一容器内兄弟节点之间的距离
     示例：tabs 底部到 input 顶部的距离
   - 内边距：容器边界到其直接子节点的距离
     示例：卡片左边界到文字节点的 12px
   同等级别判断依据：新增节点与参考节点在容器结构中的角色相同
   （同为兄弟间距，或同为容器到子节点的边距）
b. 无同等级别时，取相邻节点间距
c. 纵向间距与横向间距分开采样，不得混用
   （纵向：y 轴方向；横向：x 轴方向）

使用兜底值时必须在 run record 注明来源（采样了哪个节点、哪类间距、具体值）。
```

## 组件复用通用原则

```text
1. 内部边距
必须从采样端读取 padding 实际值应用，不得估算。

2. 内部元素间距
必须从采样端子节点实际坐标读取间距关系（左边距、右边距、元素间距）；目标容器尺寸与采样端不同时，必须用采样端的间距关系重新计算子元素位置，不得直接照抄采样端绝对坐标值。

3. 容器尺寸
- 一级容器（独占一行，左右边距固定）：不可变尺寸
- 标签、胶囊、按钮、Tab 等小控件：必须跟随内容实际尺寸计算，不得固定为采样端尺寸
- 输入框（内部文本固定宽高）：不可变尺寸
- fill/stretch 容器：跟随父级，不跟内容

4. 文字节点
Auto Layout 容器内的文字节点必须同时满足以下两项，缺一不可：
(a) textAutoResize = 'WIDTH_AND_HEIGHT'：文字节点高度由内容自然撑开，不得固定高度。
(b) layoutSizingHorizontal = FILL：文字节点横向撑满父级容器，不得写死固定宽度。
例外：有明确截断需求的业务场景（如固定行数截断、有明确 max-height 的内容区）可限定高度，但必须在 run record 写出原因和采样端对应高度值；有明确固定宽度语义的控件（如参数胶囊内文字）可保持 FIXED 宽度，同样必须写出原因。
适用范围包括：标题、字段名、字段值、按钮文案、导航文字、辅助说明，以及输入框内的占位文字、提示文字、示例文字（placeholder、example、guidance 类节点）——这类节点同样适用双重约束，不得以"父级容器有固定高度"或"输入框场景"为由豁免。输入框容器本身可以有固定高度，但容器内的文字节点本身必须满足上述双重约束。

5. 图标节点
有资产节点优先复用；不能复用时继承尺寸、颜色、位置，内部图形可字符暂代，必须保留独立 icon 语义节点，run record 写出不可复用原因。

6. 设计规范
颜色、圆角、描边、投影优先复用采样端；不能复用时从 design-system-draft.json 找 style_id，不得临场自造数值。
```

## Module 背景与内容边距规则

```text
Hi-Fi 生成阶段，凡 Module 带有视觉背景属性（fill、stroke、cornerRadius、shadow）且需要 padding 控制内容与背景边界之间的视觉间距，必须使用 inner_wrapper 结构，不得将背景属性与 padding 同时设在 Module 本身上：
- Module 本身：全宽，无背景属性，无 padding，constraints=STRETCH。
- inner_wrapper（命名如 {module}_inner 或 {module}_card）：承载背景属性（fill、stroke、cornerRadius、shadow），设置 padding，layoutSizingHorizontal=FILL。
- 子节点：横向撑满的子节点设 layoutSizingHorizontal=FILL；有固定宽度语义的控件保持 FIXED 并在 run record 说明原因。
违反此规则（背景属性与 padding 同时在 Module 上）在 Hi-Fi 生成时即判定为结构错误，必须拆分后才能继续生成。
```

## 对齐与控件文本规则

```text
同一行的图标、文字按钮和辅助操作入口必须按对应主文本或容器的视觉中心线对齐。
按钮、胶囊、Chip、标签等有背景容器的控件，文字必须在背景容器内 X/Y 双轴居中。
按钮、Tab、Segmented Control、Chip、标签、参数项、上传入口等有背景容器的控件，必须在放入 PRD 原文案和当前可用字体后重新校验文字实际占位；文字节点必须完整落在背景容器的可用区域内，不得超出背景色块、描边、圆角边界或被父级 Frame 裁切。
控件应优先继承采样源的 padding、gap、圆角、颜色、描边、高度和视觉密度；继承后若当前 PRD 文案或当前字体导致文字溢出，必须按以下顺序处理：
1. 保持 PRD 文案不变。
2. 保持文字字号、字重和行高尽量接近采样源。
3. 优先扩大背景容器宽度或文字节点宽度。
4. 必要时扩大背景容器高度或调整控件内部 Y 轴 padding。
5. 最后才允许在不改变业务层级的前提下微调同组控件间距。
不得通过改写文案、缩写文案、缩小到破坏层级的字号、隐藏溢出、遮罩裁切或把文字移出背景容器来解决溢出。
输入框占位文案的上边距和左边距应保持一致；如果 Visual Spec 指定其他内边距，必须按 Visual Spec 执行并记录原因。
当底部操作区与主输入区同属一个卡片时，在保持父容器高度不变的情况下，应优先通过调整主输入区高度来满足操作区上下间距一致。
字段名、字段值、单位 / 状态和操作图标表达不同语义时必须拆成独立节点，允许使用不同颜色、字重、视觉权重和间距；即便字段名和字段值当前使用相同字体、字号、颜色或位于同一视觉容器内，也必须因语义不同而拆分为 `{field}_label` 和 `{field}_value`，例如 `画风` 与 `二次元` 必须分开。不得为了省图层把属性类型、属性内容、单位、状态或图标合并成一个不可区分的文本。即便图标暂时用字符表达（如 ▼、●、x、+、📎），也必须保留独立的 icon / action_icon 语义节点和命名，不得并入 label 或 value 文本层。
不得为已取消下拉、切换或展开交互的字段补充箭头、chevron 或其他交互暗示图标。
凡通过 Plugin API 创建的 TEXT 节点，必须在创建后立即同时设置：(1) `textAutoResize = 'WIDTH_AND_HEIGHT'`，不得保留 Plugin API 默认的固定宽高；(2) `layoutSizingHorizontal = 'FILL'`，使文字节点横向撑满 Auto Layout 父级容器，不得写死固定宽度。业务文案节点（标题、字段名、字段值、按钮文案、导航文字等）以及输入框内的占位文字、提示文字、示例文字（placeholder、example、guidance 类节点）均适用上述双重约束。输入框容器本身可以有固定高度，但容器内的文字节点必须满足上述双重约束，不得以"父级容器固定高度"为由豁免。如果节点需要在特定容器内对齐，应通过调整节点的 x/y 坐标定位，而不是通过固定宽高裁切文字。
凡通过 Plugin API 新建的 TEXT 节点，字体 family 和 style 必须从采样端对应节点读取后显式设置，不得依赖 Figma 默认字体或在脚本中硬编码字体名称；若无对应采样端节点，必须从 design-system-draft.json 中找到对应 font_* style_id 后设置，并在 run record 中写出来源。
凡通过 Plugin API 新建的 TEXT 节点，textAlignHorizontal 必须从同组同语义的已有节点显式读取后设置，不得依赖 Figma 默认值（LEFT）。同一卡片、列表项、控件内的从属文本节点，必须与主文本节点的对齐方式一致。

## 新建 TEXT 节点强制核查清单

每次 use_figma 调用中新建 TEXT 节点后，必须在下一条消息中列出以下属性的实际核查值，未列出视为未执行：

| 属性 | 来源要求 | 实际值 |
|------|---------|-------|
| fontName（family + style） | 从同组已有节点显式读取 | — |
| fontSize | 从同组已有节点显式读取 | — |
| lineHeight | 从同组已有节点显式读取 | — |
| textAlignHorizontal | 从同组已有节点显式读取 | — |
| fills（颜色） | 从同组已有节点显式读取 | — |
| textAutoResize | 必须设为 WIDTH_AND_HEIGHT | — |

同组已有节点：同一卡片、列表项或控件内，与新建节点语义最接近的已有文字节点。
若无同组节点，从 design-system-draft.json 对应 style_id 读取，并在核查表中注明来源。
控件内同行子节点（icon、操作按钮等）的 x 坐标必须按采样端的排列关系动态计算，不得硬编码采样端的绝对 x 值。当 value 节点设置了 `textAutoResize = 'WIDTH_AND_HEIGHT'` 后宽度自然变化时，同行的 icon x 必须跟随重新计算：icon x = chip_width - right_padding - icon_width。right_padding 和 icon_width 从采样端对应节点读取（采样端 right_padding = chip_width - icon_x - icon_width）。若重新计算后 icon x ≤ value 右边界（value.x + value.width），则必须扩大 chip_width 直到 icon x > value 右边界，不得让 icon 与 value 重叠。
```

## 同行 icon/arrow 坐标动态计算规则（chip/胶囊类节点强制执行）

```text
对所有 chip/胶囊类节点内的 arrow/icon 类右侧子节点，x 坐标必须在替换 value 文案后动态计算，
禁止搬用 Visual Spec 中的固定坐标值：

Step 1：写入 value 文案
Step 2：用 get_node 读取 value 节点的实际 width
Step 3：按公式计算 icon x：
          icon.x = chip_width - right_padding - icon_width
          （right_padding 和 icon_width 从采样源节点实测，禁止估算）
Step 4（修正指导）：overlap 由 hifi_geometry gate Detection B 检出；检出后扩大 chip_width 直到 icon.x > value.x + value.width
Step 5：chip_width 被扩大后，必须级联重算同行所有后续 chip 的 x 坐标：
          后一个 chip.x = 前一个 chip.x + 扩大后 chip_width + 采样源 itemSpacing
          （itemSpacing 从采样源列表容器节点实测，禁止估算）
          若第一行总宽（所有 chip 宽度之和 + 间距之和）超出容器可用宽度，
          必须换行，并重算第二行起始 y 坐标。
Step 6：将 Step 2–5 的计算过程、overlap check 结论、最终各 chip x/width 值写入 run record。

Visual Spec 中标注的 icon x 坐标仅供参考，禁止直接用于 set_node 写入。
```

## 固定底部 CTA 规则

```text
页面主 CTA 如属于底部悬浮 / 贴底操作，不参与普通内容模块纵向间距计算。
如果一个模块或组件被 Visual Spec、命名语义或布局语义判定为悬浮、贴底、贴顶、fixed、sticky 或 overlay 显示，则该节点必须位于当前页面状态的 Z 轴最上层；不得被普通内容模块、背景层、保护层或装饰层覆盖。
贴底 / 贴顶悬浮节点应作为 Page Frame 的高优先级 overlay 子节点处理，不参与普通内容模块纵向 / 横向 Auto Layout 流；如存在保护背景层，保护背景层必须位于悬浮节点下方、普通内容上方。
固定底部 CTA 必须以 Page Frame 底部为基准定位，默认下边距为 24px；如 Visual Spec 或人工确认另有值，必须记录原因。
固定底部 CTA 在 Z 轴上必须位于页面最上层。
为避免内容与固定底部 CTA 冲突，必须在 Page Frame 底部增加一个底部保护背景层，默认高度为 90px，宽度等于 Page Frame，颜色使用页面底部背景色。
底部保护背景层必须位于固定底部 CTA 下方、普通内容模块上方；不得遮挡 CTA。
普通内容模块排布时应尽量避免关键内容进入保护渐变区；但当内容因折行、文案扩展等原因导致模块高度超出可用区域时，允许普通内容模块底部进入保护渐变区——保护渐变层的视觉作用是柔化内容与 CTA 的过渡，内容在渐变下方仍可见，不视为错误。此时不得为强制预留而压缩模块间距或 chip 内部 gap。
```

## 间距溯源验证规则

### 核心原则

```text
每一个间距值都必须有可追溯的采样源实测来源。来源断链即判定失败。
```

### 合法来源定义

```text
唯一合法来源：采样源节点的实测值。

不接受 design token 作为溯源终点。
token 值本身必须能追溯到采样源节点 ID + 实测值，否则对间距验证无效。
token 不得作为间距溯源的终点——溯源链到达 token 就停止，视为来源断链。

原因：token 表本身可能是 AI 生成或人工录入错误，
用 token 验证等于用错误验证错误，溯源链在 token 层断掉。
```

### Page 一级模块间距优先级链

```text
以下规则专用于 Page Frame 直属一级模块之间的垂直 gap
（如 top_nav → prompt_input_card、prompt_input_card → inspiration_cards 之间的间距），
不适用于组件内部间距或内边距（内边距有独立规则：直接从采样源组件节点实测）。

优先级链（三层，按序选取，不得跳层）：

Layer 1 — 复用模块自身实测间距（Level 1/2 复用时适用）
  当前模块在采样界面中有直接对应组件时，使用该组件与相邻模块的实测 y 坐标差值。
  来源格式：采样界面节点 ID + 上方模块 bottom y + 下方模块 top y + 差值 = X px。
  此层优先级最高，不得跳过。

Layer 2 — 采样界面语义相近间距（Level 3 重建且有语义匹配项时适用）
  从采样界面的一级模块间距目录中，选取结构角色最接近的值。
  间距目录须在执行前（表B）提前列出，不得执行时临场查找。

Layer 3 — 采样界面频次最高间距（Level 3 重建且无语义匹配时适用）
  使用采样界面所有一级模块间距中出现频次最高的值作为兜底。
  必须在 run record 中记录：
  「兜底原因：无语义匹配，取频次最高模块间距 = X px，来源节点 ID = 」。
  不得估算，不得使用未列入间距目录的值。
```

### 验证覆盖范围

```text
每个模块都必须验证，不区分是否发生过复用。
不得以「该模块未复用」或「该模块为 Level 3 重建」为由跳过验证。
```

需要验证的三类间距：

| 类型 | 说明 |
|------|------|
| 模块间距 | 模块与模块之间的垂直/水平间距 |
| 模块边距 | 模块相对页面或父容器的外边距 |
| 模块内边距 | 模块内部的 padding |

### 溯源记录格式

```text
每个间距值必须记录以下三项，缺一不可：

  采样源节点 ID  +  属性名（margin-top / padding-left 等）  +  实测 px 值

无法提供以上三项 → 该间距值判定为无效，不得写入 Hi-Fi 文件。
```

### 验证失败处置

```text
验证失败（来源断链）→ 打回重新采样，不得标注「人工复查」后继续推进。

唯一例外：采样源节点不可访问（节点被删除或文件权限问题）。
此情况允许标注「采样源不可达，待人工补充」，但必须明确写出不可达的节点 ID。
不得用此豁免条款掩盖「未采样」的情况。
```

### 执行位置

```text
间距溯源验证在表C（模块坐标校验表）之后、confirm 之前完成。
每个模块的三类间距溯源记录，作为表C的附属输出，随表C一同等待人工确认。
confirm 后执行时，Hi-Fi 写入的间距值必须与溯源记录完全一致，不得临场调整。
```

```text
如果线框图、PRD / PRD 补充记录、Intent、Priority Map、Layout Spec、Visual Spec 或采样来源之间发生冲突，必须停止并列出冲突项，交由人工定夺。
不得自行选择一个来源继续生成。
冲突记录必须写入 run record。
```

## 间距类属性冲突仲裁补充

```text
线框图间距（模块间距、内边距、组件间距）仅为结构示意，属于视觉表现范畴。
当线框图间距与采样源实测间距不一致时，以采样源实测值为准，不得标记为冲突项或停止等待人工裁定。

适用范围：
- 一级模块间纵向 gap（如 top_nav → prompt_input_card 之间的垂直间距）
- 页面左右 padding（内容模块相对页面边框的水平边距）
- 组件内边距（padding）、组件内子元素间距（gap）

不适用范围（仍须人工裁定的冲突）：
- 模块存在性（线框图有/无某模块）
- 模块顺序、业务语义层级、可见文案
- PRD 约束的功能入口、状态表达、控件数量

依据：hifi_generation_rules.md「旧文件视觉主导规则」第3条：
「采样端 Figma 来源 / Design System 决定视觉表现，包括…间距…」
间距属于该条款覆盖的视觉表现属性，线框图的间距值不构成约束性冲突。
```

## 高保真执行规则补丁 v1

### 一、执行前必须输出两张确认表，等待人工 confirm 后才能进入执行

**表A：复用判断表**（已有，保持不变）

**表B：页面级属性确认表**（新增）

| 属性 | 数值 | 溯源类型 | 来源节点 ID / PRD 章节 |
|------|------|----------|------------------------|
| Page Frame 背景色 | | | |
| 装饰渐变层 style_id | | | |
| 模块间距 gap | | | |
| 页面左右 padding | | | |
| clipsContent 设置 | | | |

溯源类型只允许三种：**采样端实测**（必须附节点 ID）、**PRD**（必须附章节号）、**Design System token**（必须附 style_id）。出现其他来源视为无效，停止执行。

表B 中所有间距类属性（模块间距 gap、组件内边距 padding）填写完成后，每个模块执行前必须各输出一次以下实测声明，否则不得进入该模块执行：
「我已用 use_figma 读取采样源节点 [nodeID] 的实际值：paddingTop/Bottom/Left/Right = X/X/X/X，gap = X，与表B 填写一致。」
未输出声明或声明中无节点 ID 的，视为未完成表B，不得继续。

### 二、每个模块执行拆成两步，中间有截图确认

**Step A：只做结构操作**

**Step A 执行优先级（补丁 v1.2）：**

```text
Step A 执行顺序：
1. clone 采样源节点（取视觉样式）
2. 对照线框图做 diff（取内容结构）
3. 文案、模块结构、节点数量以线框图为准
4. 视觉样式、组件风格、UI 属性以采样源为准
5. 两者有冲突时：结构和内容听线框图，样式听采样源
```

Step A 完成后截图，截图必须包含模块四边完整外边界（与本文件「逐模块截图确认 scope 规则」对齐）。人工确认结构正确后才进入 Step B。

**Step A 全量属性核验规则（补丁 v1.4，强制）：**

```text
clone 完成后，必须用 use_figma 读取采样源节点的全量属性，
与克隆节点做逐属性对比，输出核验表：

核验表格式：
| 属性 | 采样源值 | clone 实测值 | 是否一致 | 处理方式 |

核验必须覆盖以下属性，缺一不可：
- fills（含 GRADIENT_LINEAR 的完整 gradientStops 和 gradientTransform）
- strokes（含 color、weight）
- cornerRadius（每个角单独记录）
- effects（含 DROP_SHADOW 的 color、radius、offset）

核验表全部属性记录完成后，才能截图；截图通过后才能进入 Step B。
不得只核验部分属性，不得以「视觉等价」为由跳过属性记录。
```

**Step A 截图核验项补丁 v1.3（强制）：**

```text
Step A 截图核验必须同时包含以下三类检查，缺一不可：
1. 文案与结构：节点名称、文案内容、节点数量是否与线框图一致
2. 视觉样式：圆角、填充色、图标节点类型是否与采样源一致

3. 无 overlap：chip/胶囊内部各子节点边界不得互相覆盖
   （value 文字右边界必须 < arrow icon 左边界）
```

**表C：模块坐标校验表**

| 模块 | Layout Spec y | 线框图实际 y | 偏差 | 修正方式 |
|------|--------------|------------|------|---------|

**填写规则：**
- `Layout Spec y`：从 `workspace/layout_specs/` 对应页面文件中读取，必须附章节号
- `线框图实际 y`：用 `use_figma` 读取线框图帧对应模块节点的实际 `y` 属性，不得凭记忆填写
- `偏差`：`线框图实际 y - Layout Spec y`，偏差为 0 填「无」
- `修正方式`：偏差不为 0 时必须说明高保真执行时如何修正（如「高保真执行时 y 使用 Layout Spec 值」）；偏差为 0 填「继承」

**强制执行规则：**
- 表C 输出后必须停止，与表A/表B 一同等待人工回复「confirm」
- confirm 之前禁止调用任何 Figma 工具
- 高保真各模块 y 坐标必须以表C 中「Layout Spec y」为准，不得直接继承线框图 y 坐标
- 若线框图 y 与 Layout Spec y 一致，可继承；若不一致，高保真执行时必须按 Layout Spec y 重新定位
- y 坐标链偏差由 hifi_geometry gate Detection A 验证，gate FAIL 时按报错信息回修
